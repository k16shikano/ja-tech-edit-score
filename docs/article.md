# 技術書の推敲差分から「自分の選好」を取り出す試み

私は技術書の原稿を git で管理し、下書きの段階から公開できる水準まで、ブランチ上で推敲を重ねている。
LLM に下書きを書かせることはできるが、出てきた文章はそのままでは公開水準に届かない。
どこが届かないのかを言葉で挙げることはできる。
しかし、それより確かな証拠が手元にある。
下書きと推敲後の差分である。
私が原稿に何を求めているかは、直した箇所の集まりとして、すでに全部書いてあることになる。

この差分から「自分の選好」を取り出し、ソフトウェアとして使える形にできるか。
それがこの試みの主題である。
形の候補は三つある。
判定器（どちらの文章が好みかを採点する）、言語化された規則（LLM への指示に使う）、そして生成モデルへの組み込み（重みや活性化に埋め込む）だ。
結論を先に言えば、最初の二つはうまくいっており、三つ目を検証している途中である。

## 教師データとしての推敲差分

手元には 14 冊分の原稿リポジトリがある。
下書きのブランチと推敲後のブランチを突き合わせると、対応する段落の組が約 6,000 対とれる。
この組を**推敲ペア**と呼ぶ。
片方は書いた直後の文章、もう片方は私が手を入れたあとの文章で、内容はほぼ同じ、違いは「直し方」だけである。

この規模のデータで何が学べるかには、はっきりした線引きがある。
日本語をゼロから書く能力の教師としては、約 6,000 対では足りない。
文章生成の分布そのものを学ぶには、内容の多様性をカバーするだけの量が要るからだ。
一方、「二つの文章のどちらが推敲後か」を判定する教師としてなら、同じデータで足りる。
判定が学ぶのは差分の傾向だけであり、内容の多様性を丸ごと覚える必要がないからである。

## どちらが推敲後かを判定する物差し

そこでまず、判定器を作った。
凍結した日本語文埋め込みの上に線形分類器を載せ、推敲ペアの差分特徴で「どちらが推敲後か」を学習させる。
評価は書籍単位で分けた。
学習に使わなかった書籍のペアで測らないと、内容を覚えただけの分類器を高く見積もってしまうからである。

結果、この判定は初見の書籍でもペア正解率が約 0.95 に達した。
さらに Bradley-Terry 型の報酬モデル（候補一つに絶対スコアを与える形式）でも約 0.975 が出ている。
私の推敲には、書籍をまたいで一貫した傾向があり、それは小さな埋め込み空間でも線形に分離できる、ということになる。

判定ができると、生成の質を選抜で引き上げるループが組める。
LLM に候補を複数書かせ、報酬モデルで順位を付けて最良の一つを残し、改善が止まるまで繰り返す。
この方式の利点は、生成するモデルを問わないことだ。
候補がどの LLM から出てきても、採点は同じ物差しで行える。

ただし報酬モデルには癖もある。
実際に記事の推敲へ使ってみると、短い稿を強く好む傾向が見えた。
推敲ペアの中に「削る」編集が多く含まれるため、報酬が長さの短さそのものを学んでしまったのだと考えている。
選抜を機械任せで繰り返すとこのバイアスが増幅されるので、長さ比の下限のような歯止めを入れて使っている。

## 言語化された規則としての選好

判定と並行して、同じ差分を規則の文章に書き起こした。
差分の集まりを LLM に読ませ、「一文ごとに改行する」「因果を主張するときは、なぜそうなるのかの機構を一文で示す」といった文章規範として言語化させたものである。
これを LLM への指示（いわゆるスキル）としてプロンプトに与えると、フロンティアモデルの生成が目に見えて私の選好へ寄る。
現時点でいちばん品質が出る構成は、フロンティアモデルに規範を与えて書かせ、報酬モデルで選抜する、という組み合わせだ。

この形に落ち着くまでに、一つ失敗をしている。
規則ではなく、類似する推敲実例そのものをプロンプトに注入する方式を先に試した。
結果は場当たりな模倣だった。
個別の実例には、その文脈でだけ正しい直し方が含まれるため、別の文脈に当てはめると悪文になる。
この経験から、「個別実例ではなく、実例の集まりから抽象化したものを渡す」が方針になった。
規則への言語化はその一つの形であり、次に述べる二つの経路も、抽象化の形を変えた同じ発想である。

## 生成モデルの内部に埋め込む経路その1、activation steering

言語化には限界がありうる。
規則として書き出せた選好が全てなのか、言語化から漏れたが一貫している選好が残っているのか、差分を規則に起こしただけでは分からない。
そこで、言語化を経由せずに選好を生成モデルへ組み込む経路を検証し始めた。

一つ目は activation steering である。
推敲ペアの両側をオープンウェイトの LLM に通し、内部表現の平均差分ベクトルを取る。
ペアごとの内容の違いは平均で打ち消され、全ペアに共通する「推敲でどう変わるか」の方向だけが残る、という理屈だ。
このベクトルを生成時の活性化に足せば、出力が推敲後の方向へ偏ることが期待できる。

この分野の作法では、生成に介入する前に、まず「読み取り」を検証する[^repe]。
そのベクトルの方向で下書きと推敲後を分類できないなら、生成時に足しても出力を推敲後の方向へ動かせる見込みが薄いからである。

読み取りの結果は弱かった。
全トークン平均の内部表現では、初見の書籍に対する分類精度が 0.54 と、ほぼ当てずっぽうに終わった。
ペアの差分ベクトルの向きを当てる形式に変えると 0.64 まで上がり、「推敲済みか判定せよ」という指示で包んで最終トークンの表現を取ると 0.77 まで上がった（ただしこれは 64 ペアでの暫定値である）。
方向は存在するが、判定側の 0.95 には遠い。

この弱さを、私は「情報が無い」ではなく「素の表象では軸が取り出せない」と解釈している。
判定モデルを作ったときも、素の BERT 系埋め込みでは 0.5 程度しか出ず、対照学習済みの文埋め込みに替えただけで 0.9 を超えた。
生成用 LLM の素の内部表現は、このときの素の BERT と同じ立場にあるのだろう。
ただし判定のときと違い、埋め込みモデルを差し替えるという解決は使えない。
steering のベクトルは、生成に使うモデル自身の内部空間に無ければ注入できないからである。

制約もはっきりした。
このベクトルは抽出したモデル専用であり、別のモデルには持ち込めない。
API 越しのフロンティアモデルは内部に触れないので、そもそも対象外になる。
読み取りの弱さと合わせて、生成への介入は見送り、この経路は縮小した。

## 生成モデルの内部に埋め込む経路その2、編集モデルの学習

二つ目は、推敲ペアで LLM を直接学習させる経路である。
学ばせるのは「下書きを与えたら推敲後を返す」という変換であり、これを**編集モデル**と呼ぶ。
内容は入力の下書きが与えるので、モデルが学ぶのは直し方だけになる。
変換タスクの教師としてなら約 6,000 対で成立しうるし、編集タスクでは小規模モデルでも汎用 LLM に並ぶことがあるという先例もある[^coedit]。

ただし、この経路の当初の目標は下方修正した。
当初は、学習した編集モデルを「書く側」に据えて、フロンティアモデルへの依存を減らすつもりだった。
これはおそらく成立しない。
手元で学習できる規模のモデルは、直し方を学べても、日本語の運びや長文の一貫性といった基礎体力でフロンティアモデルに及ばないからだ。
フロンティア側が更新され続けるのに対し、学習済みモデルは凍結され、保守もこちら持ちになる。

それでも学習する意味は残っている。
この学習は、言語化を経由せずにペアの差分を重みへ写す。
だから、学習済みの編集モデルを「同じベースモデルに規範を指示として与えたもの」と比較すれば、規則の文章にしきれなかった何かがあるかを測定できる。
同じベース同士の比較なら基礎体力の差は混ざらず、違いは「差分を重みで学んだか、言語化された規則を読んだか」だけになる。
評価には学習に使わなかった 3 冊を確保してあり、判定はいつもの報酬モデルで行う。

この測定には、どちらに転んでも取り分がある。
編集モデルが勝てば、言語化から漏れた選好が学習可能な形で存在する証拠になる。
その場合は、同じ学習データをベンダーの Fine-tuning API に載せ、フロンティア級のモデルに選好を埋め込む経路が現実味を持つ。
負けるか同等なら、「私の選好は言語化でほぼ尽きている」という知見になり、規則と選抜に投資する現在の運用の裏付けが取れる。

現在、この学習を外部 GPU で回している最中である。

## できることとできないことの現在地

できることは三つある。

- どの LLM の出力に対しても、推敲後らしさを採点し、候補から選抜できる（生成モデル非依存）
- 言語化した規範を指示に入れることで、フロンティアモデルの生成を選好へ寄せられる
- 原稿が増えるたびに差分を採掘し、判定モデルと規範を更新し続けられる

できないこと、または見込みが薄いと分かったことも三つある。

- API 越しのモデルの内部には介入できない。steering のベクトルは抽出元のモデル専用である
- 手元で学習した編集モデルで、フロンティアモデルの生成品質を置き換えることは難しい
- 脚注の作法や見出しの付け方のような規範の個別条項を、ベクトル一本で制御することはできない。ベクトルで与えられるのは「推敲済みらしい方向への圧力」までである

## これからやること

直近は、学習中の編集モデルの評価である。
学習に使わなかった 3 冊の下書きを編集モデルに推敲させ、同じベースモデルに規範を与えた場合と、報酬モデルの採点で突き合わせる。
長さ比と編集の強さも、人間の推敲の分布と比べて監視する。
報酬の短文びいきを編集モデルが学んでいないかを確かめるためである。

その結果次第で、次の一手が分かれる。
言語化から漏れた選好の証拠が出れば、ベンダーの Fine-tuning API へ同じデータを載せる検討に進む。
出なければ、規範の改訂と選抜ループの改善に集中する。
どちらの場合も、判定する側の物差し（報酬モデル）はすべての経路の土台であり続けるので、その改善（現在は凍結埋め込みと線形ヘッドで済ませている部分の学習強化）は並行して進めるつもりである。

[^repe]: representation engineering と呼ばれる一連の研究の作法である。内部表現から概念の方向を読み取れることを確認してから、その方向で生成を操作する。

[^coedit]: Grammarly の CoEdIT が知られている。編集指示付きのデータで学習した比較的小さなモデルが、編集タスクの自動評価で汎用の大規模モデルを上回ることを報告した。
