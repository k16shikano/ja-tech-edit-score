# ja-tech-edit-scoreで何ができるようになったか

**ja-tech-edit-score**は、推敲文を生成しない。
Git上の下書きと推敲の差分から人間の選好を学習し、与えられた候補を採点する。
ここまでの作業で変わったのは、採点そのものの精度と、採点結果を選定と打ち切りに使える形にしたことである。

教師データの形は変わっていない。
人間が推敲した前後の段落ペアがおよそ5900件あり、それを教師にする。
生成モデルをゼロから育てる話ではない。

モデル比較のための評価は、書籍単位の**held-out**（leave-one-project-out）にした。
段落をランダムに分けると、同じ章の文体が訓練側に漏れて、未知の書籍への性能を過大評価しやすいからである。
この評価で埋め込みを差し替えたところ、静的埋め込みより`cl-nagoya/ruri-v3-30m`の方がよかった。
本番学習の既定埋め込みはそちらへ移した。

候補1件の「良さ」をスカラーで出す**Bradley-Terry報酬**も載せた。
これでBest-of-N（複数案から最良を選ぶ）が素直に書ける。
現版と再推敲を比べ、差が小さければ止める**収束判定**も入れた。
Best-of-Nは「どの案を採るか」だけを決める。
止まり方まで決めるわけではないので、打ち切りは別部品として置く。

いまの運用では、文章を書く本体はComposerに任せ、採点と選定と打ち切りだけをこのリポジトリが担う。
コマンドとしては、`rank`が候補を選び、`converge`が打ち切りを判定する。

クロスエンコーダ化や編集モデル自体のfine-tuneは、まだ手がけていない。
それでも、まともな日本語の候補を複数出させ、選好に近いものを選び、焼き直しが効かなくなったら止める、ところまでは動く。

これを作文力の向上と読むわけにはいかない。
選好データだけでは書く側の能力は増えない。
増えたのは選定と打ち切りである。
