# 系統1：編集モデル（SFT + on-policy 選好）

手持ちの推敲ペア（約 5,900 件）で、**下書きを推敲済みへ変換する**ローカル LLM を学習する構想。
生成品質でフロンティア＋規範スキルを置き換えることが主目標ではなく、選好の残り方を測り、必要なときだけベンダー FT へ進む実験として位置づける（後述「本系統の位置づけ」）。

関連：選好の採点は [WORKFLOW.md](WORKFLOW.md)、全体の優先順位は [ROADMAP.md](ROADMAP.md)、活性化による別経路は [ACTIVATION-STEERING.md](ACTIVATION-STEERING.md)。
高火力 DOK での SFT 手順は [DOK-EDIT-SFT.md](DOK-EDIT-SFT.md)。

## なぜこの形か

ゼロから文章を書く分布 \(P(y)\) を約 5,900 件で学ぶのは規模が足りない。
しかし編集モデルが学ぶのは条件付き分布 \(\pi(y \mid x)\)（\(x\)＝下書き、\(y\)＝推敲後）である。
内容の多くは \(x\) が与えるので、モデルが埋めるのは「直し方」に近い不確実性だけになる。
事前学習済みの日本語 LLM が流暢さを既に持つ前提なら、QLoRA 程度の更新で足りることが多い。

Grammarly の CoEdIT（指示付き編集の instruction tuning）も、編集タスクでは小規模モデルでも汎用 LLM を上回ることがある、という先例がある。

## kNN 実例注入との違い

過去に試した系統4（類似ペアをプロンプトへ実例挿入）は、場当たり的な模倣に終わりやすく、ひどい文が出やすかった。
その反省から作ったのが、差分の特徴を言語化した規範スキルである。

系統1は、個別実例を文脈に置くのではなく、全ペア上で勾配を平均する。
表面的な一件への過適合は打ち消され、コーパス全体で一貫する編集傾向が重みに残る、というのが狙いである。
規範スキルが「言語化された規則」として与えるものを、パラメータとして与える経路、という位置づけになる。

## 理論の要約

### SFT

最尤推定で \(\pi(y \mid x)\) を人間の推敲に寄せる。
複数の正しい直しがあるとき平均に寄りやすく、編集が保守的（ほぼコピー）になることがある。
また、自分の生成分布上での訓練ではないため、生成途中の逸脱を自力で戻しにくい。

### DPO と既存 BT 報酬の関係

KL 正則化付き RLHF の最適ポリシーは

\[
\pi^\*(y \mid x) \propto \pi_{\mathrm{ref}}(y \mid x)\,\exp\bigl(r(x,y)/\beta\bigr)
\]

という閉形式を持つ。
DPO は Bradley-Terry 選好モデル経由で \(r\) を消去し、chosen / rejected ペアだけでポリシーを更新する。
本リポジトリの BT 報酬（`outputs/pref-bt`）は、同じ族の報酬を**明示的に**持つ装置である。

静的に chosen＝人間推敲・rejected＝下書きだけだと、rejected が自明で信号が弱く、かつオフポリシーになりやすい。

### on-policy 化（iterative DPO / RAFT）

1. SFT 済みモデルが温度を上げて N 案を出す
2. BT 報酬で勝敗を付ける（既存の `rank` と同じ裁定）
3. 勝者だけで続ける（RAFT）か、勝者 vs 敗者で DPO する

BT の cross-project pair accuracy は既測（micro 約 0.975）なので、審査員の汎化は出発点として使える。
DeepMind の BOND は Best-of-N 分布そのものの蒸留だが、小規模では RAFT が最短接続である。

### 報酬ハッキング

記事デモでは、BT 報酬が短い稿を強く好む傾向が見えた。
iterative 最適化はこのバイアスを増幅しうる。
長さ比の下限、マージン閾値、KL アンカー、held-out での人間抜き取りを最初から入れる。

## 本系統の位置づけ

### 前提：なにが「勝てない」のか

系統1の当初の目的は、推敲ペアで学習した自前の編集モデルを「書く側」に据え、フロンティアモデルへの依存を減らすことだった。
この目的は、生成品質の比較ではおそらく成立しない。

8B級のローカルモデルは、推敲の傾向を学べても、日本語の運びや長文の一貫性といった基礎体力でフロンティアモデルに及ばないからだ。
CoEdIT のような「編集タスクなら小型でも勝てる」という先例は、狭い編集指示と自動指標での話で、本書の原稿水準の推敲と同条件ではない。
フロンティア側は今後も勝手に強くなるのに対し、FT したローカルモデルは凍結され、保守もこちら持ちになる。

「書く主役を置き換える」投資としては分が悪い。
では学習する意味が消えるかというと、そうではない。
系統1には、置き換え以外に答えられる問いが残っている。

### 意味1：言語化から漏れた選好があるかを測る

規範スキルと推敲ペアは、同じ推敲差分から作られた別の取り出し方である。
規範スキルは、差分を LLM に読ませて「一文ごとに改行する」「因果を主張するときは機構を示す」のような**規則の文章**として言語化したものだ。
一方、推敲ペアそのものには、言語化を経ていない差分がすべて残っている。

ここで問いが立つ。
「あなたの推敲には、規則として言語化できた部分しかないのか。それとも、言語化から漏れたが一貫している選好が残っているのか」。
規範スキルがよく効いている現状は、前者を示唆するようにも見える。
しかし、それを確かめた実験はまだない。

SFT はこの問いに答える実験になる。
学習は言語化を経由せず、ペアの差分を勾配としてそのまま重みに写すからだ。
比較の相手を「同じベースモデル＋規範スキル」に置くところが要点である。
フロンティアと比べると基礎体力の差が混ざってしまう。
同じベースモデル同士なら、違いは「差分を重みで学んだか、言語化された規則を指示として読んだか」だけになる。

学習済みモデルが held-out（学習に使わなかった3冊）で規範スキル付きのベースモデルに勝つなら、その差は、規則の文章にしきれなかった何かがペアから学ばれた証拠になる。
負けるか同等なら、「言語化できる部分でほぼ尽きている」という知見が得られ、それはそれで規範スキル路線の裏付けになる。

### 意味2：証拠が出たときの出口を用意する

意味1の実験で「言語化から漏れた選好が学習で載る」と分かったとしよう。
そのとき、ローカルモデルに載せて終わりでは、基礎体力の壁で生成品質が頭打ちになる。

出口は、同じ学習データをベンダーの Fine-tuning API（OpenAI などが SFT と選好学習を受け付けている）に載せることだ。
これなら、フロンティア級の基礎体力を持つモデルに、あなたの選好を重みとして埋め込める。
「フロンティアモデルに対する FT」の現実的な形はこれである。

順序が大事で、この出口は意味1の証拠が出た場合にだけ意味を持つ。
言語化で尽きているなら、規範スキルをプロンプトに入れる現行運用と結果が変わらず、FT の費用だけが残るからだ。
ローカルでの SFT 実験は、その支出判断のための安価な前哨戦、という位置づけになる。

### 生成品質だけが目的のときの優先

生成品質だけが目的なら、既存のフロンティア＋規範スキル＋Best-of-N（`rank` / `converge`）の運用強化が最も割が良い。
選好はモデルに埋め込まなくても、採点・選抜側で効かせられる。
規範スキル自体を差分コーパスから改訂し続けるのも、同じ方向の投資である。

要するに、系統1は「フロンティアに勝つ生成モデルを作る」計画から、「言語化から漏れた選好の存在を測り、あればベンダー FT でフロンティア級モデルに埋め込む」実験に役割を変えて残る。
